Billing pipeline plan

A small plan for turning Oseberg_Proff's usage counters into a durable, auditable billing pipeline that's flexible to future pricing changes. Not a commitment to build — a design document to anchor the conversation with Oseberg.

Open questions (block on these)

Confirming these with Oseberg before any code change. They determine the data model and pricing engine. The first one is the most urgent — it may invalidate invoices already issued.

Q-1
Must Is the premium {orgnr}_{yyyyMM} row-key intentional, or a bug?
The current basic-activity table keys by {domain}_{yyyyMM}. The premium-activity table keys by {orgnr}_{yyyyMM} — meaning rows are not directly scoped to a tenant. If the contract bills the tenant per Proff Premium call, the count is wrong (mixed across tenants). If the contract bills per-orgnr-lookup (e.g., "you bought tracking for 50 companies"), the key makes sense — but the tenant who looked it up isn't on the row.
Q-2
Must Is there an allowlist of orgnumbers per tenant, and if so where does it live?
Stephan's recollection: tenants have a set of orgnumbers they're allowed to look up. ProffConfiguration only holds active_subscription + premium_subscription bool flags — no allowlist visible. Either the allowlist exists in a table the audit missed, or enforcement is not implemented and any premium-licensed tenant can look up anything.
Q-3
Must What's the exact pricing structure per SKU?
Stephan's current understanding: billed on all calls; premium calls cost more than basic. Need the actual NOK/call rates per SKU, whether there are tiered volume discounts, included call quotas, overage rates. Pricing changes occasionally — the pipeline should support retroactive rate changes (e.g., backfilling a month at a new rate) and forward-dated rates (e.g., "from June 1 the basic rate goes up").
Q-4
Should Cache hits — billable or not?
Current code increments the counter on cache hits (per CHANGES.md BE-1 — "tracks tenant lookup activity for billing, not raw Proff calls"). Confirm this matches what customer contracts say. Two reasonable models: (a) bill on lookups the tenant made (current), (b) bill on raw Proff calls (gives them an incentive to optimise; reduces Oseberg's gross margin).
Q-5
Should How is the invoice currently produced?
Manual KQL or table query at month-end? An Excel template? A Power BI report bound to the table? This determines the migration path — if there's no automation today, this plan can replace it; if there is automation, we need a compatibility shim during transition.
Q-6
Could Per-tenant monthly quotas and overage handling?
Are there hard quotas ("after N calls/month you're cut off") or soft overage charges ("call N+1 onwards is at overage rate")? Currently neither is implemented — only a active_subscription boolean check. Worth deciding now so the data model carries the necessary fields, even if enforcement is deferred.
Q-7
Could Retention requirements for billable events?
Tax/accounting in Norway typically requires 5 years of supporting documentation. The event log should be retained at least that long; Azure Tables are cheap enough that this is essentially free. Confirm with Oseberg's accountant.

What we have today

Two Azure Tables in proffwebstorage track per-tenant request counts:

TableRow keyUpdated whenIssue for billing
ProffRequestActivity{domain}_{yyyyMM}Every ProffCompanyLookup call, incl. cache hitsNot idempotent; no per-request audit trail
ProffPremiumRequestActivity{orgnr}_{yyyyMM}Every ProffPremiumLookup call⚠ Key doesn't include tenant — see Q-1

Both writes happen in the request hot path. Failure modes (connection drop after counter increment, retry without idempotency, table throttle) cause counts to drift from reality over time. There's no replayable event log — you cannot rebuild the counters from an immutable source if anything goes wrong.

Target architecture

Three separations: capture, store, aggregate. Pricing is a fourth concern that operates against the stored events, not the live function.

The hot path emits one queue message per request and returns. Everything downstream is async and replayable. Pricing is applied at aggregation time, not capture time — so rate changes can be applied retroactively by re-running the aggregator over the event log.

Design principles

Append-only event log
Captured events are never updated or deleted. Aggregation reads them in bulk; corrections are emitted as new compensating events with a reason code, not mutations.
Idempotent capture
Every event carries a requestId (GUID from the function invocation). Row key in BillableEvents is {yyyyMM}_{requestId} — duplicate enqueue + drain is a no-op upsert.
Pricing is data, not code
PricingHistory is a versioned table with validFrom / validTo per SKU. Aggregator joins events against pricing valid at the event's timestamp. Rate changes are inserts, not edits.
Off the hot path
Hot-path cost is one queue message — sub-10ms, fire-and-forget. A function instance crashing after the queue enqueue but before responding is the only failure mode; everything else replays from the queue.
Aggregation is pure
Given the event log + pricing table at a point in time, the invoice is deterministic. Re-running the aggregator over the same inputs always produces the same outputs — easy to audit, easy to dry-run.
Multi-SKU from day one
Event schema carries sku as a free-form string (basic, premium, future premium-credit, nordic-extension, etc.). New SKUs require only a row in PricingHistory, no code change.

Data model

Event schema (BillableEvents)

PartitionKey: {tenant}                  // domain string, e.g. "suran.crm4.dynamics.com"
RowKey:       {yyyyMM}_{requestId}      // sortable by month, unique per call

// Body fields
requestId           : string (GUID)     // same as suffix in RowKey, for joining
timestamp           : DateTimeOffset    // UTC, when the request was served
endpoint            : string            // "ProffCompanyLookup" | "ProffPremiumLookup"
queryType           : string            // "search" | "enrich" | "premium-credit"
organisationNumber  : string?           // when applicable
sku                 : string            // "basic" | "premium" — billable category
cacheStatus         : string            // "hit" | "miss" | "stale" — informational
upstreamCalled      : bool              // did we actually hit Proff? — informational
responseStatus      : int               // HTTP status returned to caller
billable            : bool              // false on early-failure (no-license denials etc.)
notes               : string?           // free-form, e.g. "denied — no premium license"

Pricing schema (PricingHistory)

PartitionKey: {sku}                     // "basic", "premium", etc.
RowKey:       {yyyyMMddTHHmmss}_validFrom  // inverse-sortable

// Body fields
sku             : string
validFrom       : DateTimeOffset
validTo         : DateTimeOffset?       // null = currently active
ratePerCall     : decimal               // NOK
currency        : string                // "NOK"
includedQuota   : int?                  // calls included per month at this rate (null = unlimited)
overageRate     : decimal?              // NOK per call beyond quota
notes           : string?               // reason for change, ticket ref, etc.

Invoice line schema (InvoiceLines)

PartitionKey: {tenant}
RowKey:       {yyyyMM}_{sku}

// Body fields
tenant          : string
month           : string                // "2026-05"
sku             : string
calls           : long                  // total billable calls in month for this SKU
includedQuota   : int?
overageCalls    : long                  // max(0, calls - includedQuota)
baseAmount      : decimal               // (calls within quota) × ratePerCall
overageAmount   : decimal               // overageCalls × overageRate
totalAmount     : decimal
currency        : string
generatedAt     : DateTimeOffset        // when the aggregator ran
sourceVersion   : string                // git SHA of the aggregator that ran
Why these shapes

PartitionKey by tenant on BillableEvents means per-tenant aggregation is a single partition scan — fast and cheap. RowKey starts with month so the aggregator can fetch one month for one tenant as a contiguous range query. InvoiceLines is denormalised on purpose: each row is a finished, attributable bill line; you can recompute by re-running the aggregator, but the row itself is the canonical answer.

Pricing model — flexibility cases

Concrete scenarios the schema supports without code changes:

ScenarioHow it's handled
Price increase on June 1, 2026 New row in PricingHistory: sku=basic, validFrom=2026-06-01, ratePerCall=0.60. Set validTo on the previous row. Events in May continue to use the old rate; June onwards uses the new rate.
Retroactive rate correction (e.g., "the May rate was wrong, here's the correct one") Edit the historical row (or insert a corrected one and mark the old as superseded). Re-run the aggregator for May; InvoiceLines rows get a new generatedAt. Old PDF invoices stand; recompute is auditable.
Per-tenant negotiated rate Add an optional tenant field on PricingHistory (null = applies to all). Aggregator prefers tenant-specific rate when present.
New SKU (e.g., "premium-credit-watch") Insert a row in PricingHistory for the new SKU. Update the function to emit events with that sku value. No aggregator change.
Volume tier (first 1000 calls free, next 9000 @ 0.50 NOK, rest @ 0.30 NOK) Slightly more complex — extend PricingHistory with a tiers field (JSON array of { fromCall, ratePerCall }) and teach the aggregator to walk the tiers. Add when needed, not on day one.
Cache-hit-billable toggle changed mid-month Aggregator filters on billable=true. Toggle the convention in the function and re-run aggregator. Hot path event still captured even for non-billable cases — useful for audit even if not invoiced.

Phased migration

Six phases. Phase 1–2 are zero-risk additions; phase 3 onwards is when the live billing data shifts. Each phase ships independently.

1
Confirm Q-1 through Q-5 Block on these. The premium-key issue (Q-1) might require backfilling or compensating invoices already issued — handle that before any architectural work.
prereq
2
Stand up the queue, drain, and event-log tables in osb-test Create Storage Queue, BillableEvents table, drain function. No production traffic yet. Idempotency test via deliberate duplicate enqueues.
S · 1 day
3
Add event emission to the function — shadow mode Triggers enqueue billable events on every request, but the existing ProffRequestActivity / ProffPremiumRequestActivity writes continue unchanged. Run for 2–4 weeks; compare aggregator output against the legacy counters; reconcile differences. Lands as a feature flag.
M · ~3 days
4
Build the aggregator + pricing tables TimerTrigger function reads BillableEvents for the previous calendar month, joins against PricingHistory, writes InvoiceLines. Idempotent — re-running over the same month produces the same output. Add a dry-run mode that writes to a separate table for review before going live.
M · ~3 days
5
Switch invoicing to read from InvoiceLines Whatever consumes the current activity tables (Q-5 answer) gets pointed at the new table. Old tables continue writing in parallel for one cycle as a safety net.
S–M · depends on Q-5
6
Decommission old activity tables Remove the in-line counter writes from the function. Mark the two old tables read-only for 1 month; then drop them. BillableEvents is now the only source of billing truth.
S · half a day

Total effort: ~7–10 days of focused work after Q-1–5 are answered. Best done as a separate workstream from the .NET 10 upgrade; ship it on whichever function app survives the cutover.

Things to decide later (parking lot)

These don't block the plan but should be revisited once it's live:

  • Customer-facing usage dashboard — see RM-OBS-2. The InvoiceLines table makes this trivial. Each customer gets a read-only view of their own rows.
  • Quota enforcement (Q-6) — once the data model carries includedQuota and overageRate, enforcing a hard cutoff is a small read against the current month's InvoiceLines before the function returns.
  • Webhook / Stripe integration — if Oseberg ever wants automatic invoice generation, the InvoiceLines table is the natural integration point. Each finalised row could trigger an outbound webhook.
  • Allowlist-of-orgnumbers enforcement (Q-2) — if confirmed as a contract term, add a check between license validation and the cache lookup. Reject with a billable=false event for audit trail.
  • Multi-currency — schema already carries currency, but the aggregator doesn't yet handle conversion. Defer until there's a customer that needs it.
  • Backfilling old months — once the aggregator works, an opt-in script could re-derive historical invoices from the legacy ProffRequestActivity data and write them to InvoiceLines. Useful for trend analysis. Acknowledges that old data has the premium-keying bug from Q-1 — flag those rows as reconstructed, not captured.
