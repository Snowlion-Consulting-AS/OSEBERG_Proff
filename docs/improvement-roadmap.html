Backlog

Effort Area

Caching

RM-CACHE-1

Explicit cachedAt + 90-day TTL on premium cache

S Cache

The current ProffPremiumCache dedupes correctly but never expires — after 3 months, you serve stale credit ratings indefinitely. This is the missing half of the BE-2 fix from Phase 1.

What: Add a cachedAt: DateTimeOffset column on write; on read, return null if older than 90 days (fall through to Proff, write-through refreshes the row). Missing field → treat as stale → self-heals.

Dependencies
None — independent change. Can ship in any release window.
Tied to
Companion: Risk register mentions this as a Phase-2 caveat under BE-2.
RM-CACHE-2

Per-entry override using Proff's own "modified" field

S Cache

If Proff surfaces a "last updated" timestamp on the credit-rating record itself, trust Proff's freshness signal over our cache-age heuristic. Lower false-stale rate; fewer wasted refreshes.

Investigation needed — confirm what Proff returns. CreditRating already deserialises an Updated: DateTime field, but it's unclear whether that's the rating's recompute timestamp or just the API response time. Check Proff Premium API docs and a couple of live responses.
Dependencies
RM-CACHE-1 (cachedAt) — this overrides it for newer entries
Decision rule
If Updated within last 90 days → serve cached. Else → refresh.
RM-CACHE-3

Negative caching for missing org numbers

S Cache

When Proff returns no data for an orgnr (newly registered, deregistered, typo), the current code writes nothing — so the next user re-asks Proff. Negative caching dampens this repeat-miss pattern.

What: Store a sentinel row with isMiss = true and a shorter TTL (1–7 days). On read, treat as cached "no data" within the window; expire faster than positive cache because new companies do appear.

Dependencies
Best paired with RM-CACHE-1 (already changing the read path)
Watch out for
Don't negative-cache transient Proff outages (5xx). Only cache 200 + empty payload.
RM-CACHE-4

Proactive cache refresh (timer-triggered)

M Cache

User-facing lookups never pay the Proff round-trip if soon-to-expire entries are refreshed in the background. Hides Proff latency entirely for hot orgnrs.

What: New TimerTrigger function runs nightly, scans ProffPremiumCache for entries with cachedAt between 80 and 90 days old, refreshes them via Proff. Bounded fetch budget per run to avoid spend spikes.

Dependencies
RM-CACHE-1 (need cachedAt to find candidates)
Budget guard
Cap refreshes per night (e.g., 200) and skip if any tenant is over quota.
Cost note
This adds Proff calls — only worth it if cold credit-rating latency hurts UX or if you want to amortise refresh spend across off-peak hours.
RM-CACHE-5

Cache stampede protection

M Cache

Two concurrent requests for the same expired orgnr both miss the cache → both hit Proff → both write back. Cheap problem at current scale, expensive if scale grows.

Investigation needed — measure actual concurrent-miss rate via App Insights custom dimension on cache decisions (depends on RM-OBS-1). If < 1%, skip this entirely.

Options if measured high:

  • Lock entry in cache: write a "refreshing" sentinel with short TTL on miss; concurrent callers see it and either wait or return previous-stale.
  • Single-flight in-process: a per-worker ConcurrentDictionary<string, Task> so concurrent calls within one worker share one Proff fetch. Doesn't help cross-instance.
  • Distributed lock via Azure Blob lease: stronger but adds latency to every cache miss. Only if scale-out becomes load-bearing.
RM-CACHE-6

Distributed cache for short-lived search/enrich data

L Cache

The current IMemoryCache is per-worker. Once Azure scales out to N instances, hit rate drops to ~1/N for the same query stream. Distributed cache fixes it.

Three pragmatic options, in order of operational weight:

  1. APIM response cache (no Redis). If you adopt RM-ARCH-1, the gateway caches by URL+headers shared across all workers — free. Best fit.
  2. Azure Cache for Redis Basic tier (~kr 150–500/month). Swap IMemoryCache for IDistributedCache. Single dependency, well-understood, but a service to monitor and pay for.
  3. Azure Table as cheap distributed cache. Same shape as ProffPremiumCache already. Works, but adds latency vs Redis on every cache check.
Triggered by
Sustained scale-out (Premium plan, multiple Always Ready instances) or measurable cross-instance hit-rate drop
Don't do this yet
Current Phase 1 work targets a single Consumption instance with a low call volume — per-worker cache is fine.

Observability

RM-OBS-1

Custom App Insights metrics for cache + Proff

S Obs

Today you can't answer "what's our cache hit rate?" or "which tenant is consuming most of the Proff budget?" without writing custom KQL against request logs. Custom metrics make these first-class.

Metrics to emit:

  • proff.cache.decision with dimensions { endpoint, decision: hit|miss|stale, tenant }
  • proff.outbound.latency_ms with dimension { endpoint, status }
  • proff.activity.count with dimension { tenant, sku: basic|premium } — eventually replaces RM-DATA-2
Dependencies
Adopting the OpenTelemetry path from the .NET 10 migration makes this trivial — emit via Meter.
Quick win
Unblocks RM-CACHE-5 by giving you the data to decide whether stampede protection is worth building.
RM-OBS-2

Per-tenant dashboards (Workbook)

M Obs

Customers periodically ask "how many lookups did we make last month?". Today the answer requires you to KQL the activity table. A parameterised Azure Workbook lets either internal support or the customer themselves answer.

What: One Workbook with a tenant-domain dropdown showing: requests/day, cache hit %, top-N searched orgnrs, error rate. Each customer gets a read-only link scoped by parameter.

Dependencies
RM-OBS-1 (custom metrics tagged with tenant)
Bonus
Workbook can be exported as PDF for invoice attachments if Oseberg ever bills by usage.
RM-OBS-3

Alerting for Proff outages and quota spikes

S Obs

You discovered the 17k/day call spike weeks after it started. A simple "outbound calls to api.proff.no > threshold" alert catches the next one in minutes.

Alerts to wire up:

  • Proff 5xx rate > 5% over 15 min → Snowlion on-call + Oseberg
  • Outbound calls to api.proff.no per hour above rolling-7-day P90 × 2 → Slack notification
  • Single tenant requests > 1000/hour → likely runaway loop on the form
Dependencies
None — works on existing App Insights data

Architecture evolution

RM-ARCH-1

API Management in front of the function

L Arch

Auth, quota, rate limiting, and response caching are all undifferentiated heavy lifting currently done in custom .NET. APIM solves all four in policy. The function becomes a thin Proff client (~150 lines).

What you get:

  • Function key disappears — APIM validates a Dataverse user JWT inbound; no more secret in the PCF bundle
  • Per-tenant rate limiting via rate-limit-by-key on the tenant claim
  • Response cache shared across all function instances (15-min cache-lookup / cache-store)
  • License/SKU enforcement moves to APIM products + subscription keys — no more ProffConfiguration table read on the hot path
  • ~50% less code in the function
Cost
APIM Consumption tier: $0.042/10k calls, first 1M/month free. For Oseberg's volume: ~kr 0/month indefinitely.
Dependencies
None technically, but easier to land after .NET 10 + Phase 1 are stable
Cutover risk
Coordinate PCF URL change with the gateway go-live. Run both in parallel for a week, then retire the direct function endpoint.
Don't do this if
The team isn't going to learn APIM policy XML. The value evaporates if it becomes a black box.
RM-ARCH-2

Async activity counting via Storage Queue

S Arch

Today every request blocks on two table writes (activity counter upsert + read). Moving it off the hot path drops user-facing latency and removes a failure class. Also a prerequisite for the billing pipeline — see the dedicated plan.

What: HTTP trigger enqueues a { tenant, endpoint, timestamp, requestId, sku, cacheStatus } message and returns. A separate QueueTrigger function drains and appends to an event log (BillableEvents).

Dependencies
None for the queue mechanic itself — but the full billing pipeline depends on this and is designed in billing-pipeline-plan.html
Cost
Storage Queue is essentially free at this volume (kr < 1/month)
Note
This is intentionally framed broader than the original draft — the activity tables underpin Oseberg's invoicing, not just ops tracking
RM-ARCH-3

Unify ProffCompanyLookup + ProffPremiumLookup

S Arch

The two trigger classes are ~80% duplicated. One trigger with an ?sku=basic|premium param removes copy-paste drift and halves the surface area to maintain.

What: Single HTTP trigger routes on a query parameter. Caching strategy chosen per SKU (memory cache for basic, table cache for premium). PCF only needs to know one URL.

Best done with
The .NET 10 port — fresh project, no migration regret
Risk
Low. The shapes are already aligned through InputParams.
RM-ARCH-4

Auth modernisation (drop function key entirely)

L Arch

Function keys in PCF bundles are a chronic leak risk. CRM-user-token-based auth (Easy Auth or APIM JWT validation) ties access to the logged-in D365 user, removes the bundled secret, and gives you audit-grade attribution per call.

Two paths to get there:

  • With APIM (RM-ARCH-1): APIM's validate-jwt against the Dataverse OAuth tenant. Clean. Recommended.
  • Without APIM: Easy Auth on the function app directly. Works but you lose the gateway-level caching/rate-limiting.
Dependencies
Ideally RM-ARCH-1 first
Customer ask
Likely required eventually for security review / ISO27001 / customer compliance audit. Pre-empt by ~6 months.

Frontend

RM-FE-1

React 16.8.6 → 17.0.2 upgrade (PCF max)

S FE

React 16.8.6 is 5+ years old. PCF's platform-library system caps at React 17.0.2 — going higher isn't supported. The 16→17 bump is small but worth doing for the new JSX transform, gradual upgrade primitives, and to land on the highest supported version while it stays patched.

PCF constraint — the platform-library manifest declaration only accepts React 16.8.6 or 17.0.2. React 18 is not available on the PCF runtime; useId, useTransition, useDeferredValue and createRoot remain off-limits. Plan accordingly: keep the hand-rolled debounce + AbortController pattern.

What: Bump React + react-dom to 17.0.2 in package.json. Update platform-library version in ControlManifest.Input.xml to 17.0.2. The 16→17 jump has no developer-facing breaking changes for typical hooks-based code.

Dependencies
None — frontend-only release
Worth knowing
Fluent UI v8 is compatible with both 16 and 17. No UI library bump needed.
Realistic value
Smaller than originally drafted. Mainly: closer to the supported ceiling, picks up the new JSX transform, opens the door to event-delegation changes some libraries depend on. Not transformative.
RM-FE-2

User-visible error UX

S FE

Today, fetch failures go to console.error. Users see empty results indistinguishable from "no matches", so they retry — burning more Proff calls. Showing a banner instead breaks the loop.

What: A Fluent UI MessageBar above the result area on fetch failure with retry hint. Pair with ErrorBoundary wrapping SearchComponent so any render exception shows "Lookup unavailable" instead of a blank PCF.

Dependencies
None
Maps to
Risk register MED-4 + LOW-5
RM-FE-3

Accessibility pass

M FE

Result cards aren't keyboard-navigable, the SearchBox has no aria-describedby, dialog has no live-region for status. Some Oseberg customers (public sector, larger enterprises) require WCAG AA — this is a "yes/no" gate for them.

What: Make result cards role="option" with tabIndex and arrow-key navigation. Wire ARIA labels on SearchBox, Dropdown, Dialog. Add alt text or aria-hidden on flag SVGs. Test with VoiceOver and NVDA.

Dependencies
Best done after RM-FE-1 (React 18 has better a11y primitives)
Maps to
Risk register LOW-4
RM-FE-4

Request timeout + loading states

S FE

Function cold start can take 3–5 seconds. There's no spinner, no timeout. Users assume nothing happened and retype, multiplying calls.

What: Add 15s AbortController timeout in handleSearch. Show a Fluent UI Spinner in the result area while in-flight. Show "Still searching…" if > 3s.

Dependencies
None
Maps to
Risk register MED-5

Operations

RM-OPS-1

Self-service tenant provisioning UI

M Ops

Adding a new customer today means manually inserting a row in the ProffConfiguration table. Error-prone, manual, and hidden from Oseberg's customer team.

Two flavours, pick one:

  • Lightweight: A small Power Apps canvas app over the Azure Table — Oseberg's team toggles flags themselves. Half a day.
  • Heavier: A Power Automate flow triggered from a Dataverse "Customer" record. Couples provisioning to the existing CRM workflow.
Dependencies
None
Worth doing if
Oseberg adds > 1 customer per month
RM-OPS-2

Backup of ProffConfiguration

S Ops

If proffwebstorage is accidentally deleted or corrupted, every tenant loses service simultaneously. The config table is small but load-bearing — a daily snapshot to a separate storage account costs cents.

What: Nightly timer-triggered function exports ProffConfiguration rows to a backup container (different account, ideally different region). 30-day retention.

Dependencies
None
Recovery test
Document the restore steps and run a tabletop exercise once a quarter.
RM-OPS-3

Infra-as-code for the function apps

M Ops

Three function apps in dev-crm-rg were created click-ops. Bicep or Terraform makes spinning up the .NET 10 replacement repeatable, reviewable, and recoverable.

What: Bicep template covering: function app, consumption plan, App Insights, storage account, all app settings, role assignments. Output a parameterised module that can stand up another environment in 5 minutes.

Dependencies
None, but ideal time is during the .NET 10 / new-app exercise
Don't try to retrofit
Existing apps — too much drift. Document the new one as IaC from day one.

Data model cleanup

RM-DATA-1

Consolidate ProffRequestActivity + ProffPremiumRequestActivity

S Data

Two activity tables with near-identical schema for two endpoints. One table with a sku column does the job and removes a code-duplication source.

What: Single ProffActivity table with row key {key}_{yyyyMM} where key is the tenant or orgnr depending on context, plus a sku field (basic|premium). One ActivityService class.

Dependencies
None functionally, but coordinate with any tenant-facing reports that read the old tables directly
Migration
One-time copy script; old tables retained read-only for 1 month.
RM-DATA-2

Replace activity tables with App Insights custom metrics

M Data

Activity counts are observability data, not business data — they don't need their own persistence layer. App Insights stores them, gives you free KQL queries, retention, and pretty dashboards.

Investigation needed — confirm Oseberg's invoicing / customer-reporting process doesn't directly query the activity tables. If a Power BI report or invoice template reads them, that integration needs porting first.
Dependencies
RM-OBS-1 (custom metric emission) + RM-DATA-1 if not already done
Payoff
Removes 2 tables, removes a hot-path table write per request, removes a class of throttling failures

Code quality

RM-QUAL-1

Unit tests for services + integration tests for triggers

M Qual

There's no test project. Phase 1 fixes had to be eyeball-verified. Every future change carries the same regression risk.

Suggested split:

  • Unit (xUnit + Moq): CompanyDataService.ConvertJArrayToCompanyDataList, InputParams validation, ProffActivityService month-rollover logic, cache-decision logic
  • Integration: Function host run + Azurite (local Azure Table emulator) + WireMock for Proff — covers the wire shape end-to-end
Dependencies
None — ideal companion to the .NET 10 port
First test to write
The cache-stale logic in RM-CACHE-1 — small, isolated, prevents the exact regression that motivated this whole roadmap.
RM-QUAL-2

JSON snapshot tests for HTTP responses

S Qual

The biggest risk in the .NET 10 + Newtonsoft → STJ migration is silent JSON-shape drift breaking the PCF. Snapshot tests catch it in CI before deploy.

What: Verify-style snapshot tests of the two HTTP responses for representative inputs. Failing diff = "you changed the PCF contract"; reviewer decides if intentional.

Library
Verify.Xunit — minimal ceremony, diff-tool integration
Coverage target
2 endpoints × 3 representative payloads = 6 snapshots. Maintainable.
RM-QUAL-3

CI/CD pipeline (Azure DevOps or GitHub Actions)

M Qual

Today, deploys are local dotnet publish → manual zip → portal upload. The release process is in someone's head and easy to get wrong. A pipeline removes the bus factor.

Minimal pipeline: build → run tests → deploy to osb-test on PR-to-main → manual approval → promote to LIVE. PCF: pcf-scripts build → solution pack → solution import via PAC CLI.

Dependencies
RM-QUAL-1 + RM-QUAL-2 for the pipeline to actually catch things
Platform
Azure DevOps fits Oseberg's existing tenant (no extra account, group SSO). GitHub Actions is fine too if the repo stays on github.com/Oseberg-Solutions.

Suggested sequencing

None of this is mandatory. If you only do four things from this doc, do these in this order:

1
RM-CACHE-1 — explicit cachedAt + 90-day TTLSmallest possible change, fixes the actual cache bug the user articulated. Bundle with .NET 10 release.
2
RM-OBS-1 — custom metricsTiny add when you adopt OpenTelemetry in the .NET 10 port. Unblocks future decisions (RM-CACHE-5, RM-OBS-2).
3
RM-CACHE-3 + RM-CACHE-2 — negative caching + Proff "Updated" overrideBundled with RM-CACHE-1, same code path. Need to confirm RM-CACHE-2's investigation before doing it.
4
RM-FE-2 + RM-FE-4 — error UX + request timeoutCloses the user-experience loop that drives retry-induced call amplification. Pair with the next PCF release.

Everything else is bandwidth-dependent. The architectural items (APIM, auth, distributed cache) only earn their keep if call volume, customer count, or compliance requirements push them onto the critical path.
