OSEBERG_Proff codebase overview

A single-page audit of the PCF + Azure Function integration that powers Proff.no company lookup inside Oseberg's customers' Dynamics 365 environments. Use this as the briefing document before the new function is provisioned on .NET 10.

Overview

The system is a thin, end-to-end vertical: a custom PCF control on the D365 Account form, a small Azure Function that proxies and caches Proff.no's API, and four Azure Tables for multi-tenant license/quota state and one premium cache. There is no message bus, no background processing, no auth besides a function key.

How a typical request fans out

Search path

User types in PCF → debounce 1s → AbortController cancels previous → GET /api/ProffCompanyLookup?query=... → tenant license check → memory-cache check → Proff list-search → JArray → CompanyData[] → activity counter +1 → JSON response → render cards.

Enrich path

User clicks a card → GET ...?organisationNumber=... → tenant license check → memory-cache check (key enrich|NO|{orgnr}) → Proff register endpoint → field remap → JSON response → PCF caches into private _* fields → notifyOutputChanged() → 25 D365 fields populated.

.NET 10 — why now

Your .NET 8 isolated worker runs out of Microsoft support on November 10, 2026 — about six months from today. Two things happen on that same day:

  • .NET 8 itself reaches end-of-support (LTS sunset)
  • The Azure Functions in-process model is killed entirely (doesn't apply to you — you're already on isolated, but worth noting)

.NET 9 (STS) ends support the same month, so picking 9 means doing this whole exercise twice. .NET 10 LTS is the right target: supported until November 2028, GA on Azure Functions isolated since Feb 19, 2026, and you're standing up new infrastructure anyway.

Do NOT use the Azure Monitor Distro inside the worker

It bundles AspNetCoreInstrumentation, which double-counts HTTP requests against the Functions host's own request telemetry. Use Azure.Monitor.OpenTelemetry.Exporter (not the …Distro package).

Breaking changes you actually have to think about

The non-events

  • HostBuilder().ConfigureFunctionsWebApplication() works unchanged on .NET 10.
  • <FrameworkReference Include="Microsoft.AspNetCore.App" /> is still the correct pattern — keep it.
  • <AzureFunctionsVersion>v4</AzureFunctionsVersion> stays at v4 (the host is independent of the worker .NET version).
  • services.AddMemoryCache(): no API change. Your Phase 1 cache work moves as-is.
  • ASP.NET Core integration HTTP triggers (HttpRequest/IActionResult): no breaking changes.

The optional new path

You can switch the bootstrap to FunctionsApplication.CreateBuilder(args) which returns an IHostApplicationBuilder — cleaner, matches modern ASP.NET Core minimal hosting. Requires Worker 2.x. Both APIs are supported side-by-side; pick the new one for the new function.

Newtonsoft considerations (the one likely to bite)

The default serializer in the isolated worker has always been System.Text.Json. Unless you wired Newtonsoft into WorkerOptions.Serializer, your function already serialises via STJ. But the defaults differ in ways your PCF consumer can choke on:

BehaviorNewtonsoft defaultSTJ default
Property namingPascalCasecamelCase
Quoted numbers ("123" → int)PermissiveStrict — throws unless NumberHandling.AllowReadingFromString
NaN / Infinity floatsAllowedThrows
Dictionary keysAs-writtenCamelcased
TimeSpanISO 8601 durationConstant "hh:mm:ss"

Three pragmatic choices:

  1. Keep Newtonsoft for HTTP triggers via services.AddMvc().AddNewtonsoftJson(). Lowest risk; zero PCF change.
  2. Match STJ to Newtonsoft defaults: PropertyNamingPolicy = null to preserve PascalCase shape the PCF expects today.
  3. Move PCF to camelCase: cleanest long-term, but requires a PCF release. Park this for a later phase.

Snapshot-test the JSON output of both functions before and after the bump. Two endpoints, ten minutes of work, complete protection.

Known gotchas (May 2026)

⚠ App Insights 3.x

Microsoft.ApplicationInsights.WorkerService 3.0/3.1/3.1.1 throws TypeLoadException against the Functions worker. Microsoft has stated no fix is planned. Pin at 2.23.0 or migrate fully to OpenTelemetry.

⚠ Exit code 150 on deploy

Multiple users on Flex Consumption with .NET 10 + Worker 2.51.0 + Sdk 2.0.7 reporting dotnet exited with code 150 (0x96). Cause: function.deps.json referencing net8.0 runtime artifacts. Clean publish, don't mix SDKs in CI.

⚠ Linux Consumption

Azure CLI will let you set .NET 10 on Linux Consumption, but the app won't start. Use Flex (recommended for new apps) or Premium on Linux. Windows Consumption is fine.

⚠ Local debugging

local.settings.json needs APPLICATIONINSIGHTS_CONNECTION_STRING set even for local runs once on the OTel path. A dummy InstrumentationKey=00000000-0000-0000-0000-000000000000 works.

⚠ Middleware response headers

Headers set after await next(context) can be lost — response may already be flushed. Set on context.GetInvocationResult().Value before return.

⚠ HttpClient lifetime

Your existing static HttpClient with per-request header mutation is a known footgun (see risk register CRIT-2/CRIT-3). Fix during the upgrade via services.AddHttpClient<T>().

Recommended Program.cs for the new function

using Microsoft.Azure.Functions.Worker.Builder;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Logging;
using OpenTelemetry.Logs;
using OpenTelemetry;

var builder = FunctionsApplication.CreateBuilder(args);

builder.ConfigureFunctionsWebApplication();

// State
builder.Services.AddMemoryCache();

// HttpClient via factory — replaces the static HttpClient footgun
builder.Services.AddHttpClient<IProffClient, ProffClient>(c =>
{
    c.BaseAddress = new Uri("https://api.proff.no/");
    c.Timeout = TimeSpan.FromSeconds(15);
});
builder.Services.AddHttpClient<IProffPremiumClient, ProffPremiumClient>(c =>
{
    c.BaseAddress = new Uri("https://ppc.proff.no/");
    c.Timeout = TimeSpan.FromSeconds(15);
});

// Per-table-storage-name singletons (avoids the per-request `new` pattern)
builder.Services.AddSingleton<IConfigurationTable>(_ => new AzureTable("ProffConfiguration"));
builder.Services.AddSingleton<IRequestActivityTable>(_ => new AzureTable("ProffRequestActivity"));
builder.Services.AddSingleton<IPremiumCacheTable>(_ => new AzureTable("ProffPremiumCache"));
builder.Services.AddSingleton<IPremiumActivityTable>(_ => new AzureTable("ProffPremiumRequestActivity"));

// Telemetry — OpenTelemetry path, not the broken classic AI 3.x
builder.Services
    .AddOpenTelemetry()
    .UseFunctionsWorkerDefaults()
    .UseAzureMonitorExporter();

builder.Logging.AddOpenTelemetry(b => b.IncludeScopes = true);

// JSON: preserve PascalCase to keep the PCF consumer working unchanged
builder.Services.AddMvc().AddJsonOptions(o =>
{
    o.JsonSerializerOptions.PropertyNamingPolicy = null;
});

// Fail fast on missing config
foreach (var key in new[] { "PROFF_API_KEY", "PROFF_PREMIUM_API_TOKEN", "AZURE_PROFF_WEBSTORAGE_CONNECTIONSTRING" })
    if (string.IsNullOrWhiteSpace(Environment.GetEnvironmentVariable(key)))
        throw new InvalidOperationException($"Missing app setting: {key}");

builder.Build().Run();

Risk register

18 items found in the active codebase. Severity-coded. Filter by area:

  • CRIT-0
    Function API key committed in source Critical PCF
    Production Azure Function key is hard-coded in config.ts and present in git history. Anyone with repo read access can call the function.
    VirtualControl/VirtualControl/config/config.ts:4
    Fix: Rotate the key now (regardless of this upgrade). Move loading to environment-based config injection at build time, or front the function with Easy Auth / a CRM-authenticated proxy.
  • CRIT-1
    Shared static HttpClient with per-request header mutation Critical Backend
    ProffApiService.httpClient is private static readonly but DefaultRequestHeaders.Authorization is reassigned on each call. Concurrent invocations race on the header, producing intermittent 401s or cross-tenant headers.
    LIVE-ProffCompanyLookup/Services/API/ProffApiService.cs:9
    Fix: Switch to IHttpClientFactory via services.AddHttpClient<T>(), set the header per-request on a HttpRequestMessage, not on the shared client.
  • CRIT-2
    Static HttpClient field re-instantiated per object Critical Backend
    ProffPremiumApiService._httpClient is declared static but reassigned new HttpClient() in the constructor. Result: socket exhaustion over time, no DNS rotation. Worse: each ProffPremiumLookup.Run() currently constructs a new service per request.
    LIVE-ProffCompanyLookup/Services/API/ProffPremiumApiService.cs:12,21
    Fix: Replace with IHttpClientFactory + typed client.
  • CRIT-3
    Static HttpResponseData _response in trigger classes Critical Backend
    Both ProffCompanyLookup and ProffPremiumLookup hold private static HttpResponseData? _response. Two concurrent invocations on the same worker can clobber each other's response object.
    LIVE-ProffCompanyLookup/ProffCompanyLookup.cs:28 · LIVE-ProffCompanyLookup/ProffPremiumLookup.cs:23
    Fix: Make local to the Run method. HttpHelper.ConstructHttpResponse currently takes it as a parameter but always overwrites — drop the parameter entirely.
  • HIGH-1
    Stateful _entity cache on AzureTableStorageService High Backend
    RetrieveEntityAsync writes to a shared _entity instance field. Two concurrent calls can race; callers receive stale or wrong entities. On a miss, the method returns the cached previous entity instead of null.
    LIVE-ProffCompanyLookup/Infrastructure/AzureTableStorageService.cs:11,20-26
    Fix: Return the entity directly; drop the instance field. Make the service stateless and register as singleton per-table.
  • HIGH-2
    No startup validation of secrets High Backend
    PROFF_API_KEY, PROFF_PREMIUM_API_TOKEN, AZURE_PROFF_WEBSTORAGE_CONNECTIONSTRING are read at construction with no null check. A misconfigured environment fails silently (malformed auth header) or only on first table operation.
    ProffApiService.cs:16 · ProffPremiumApiService.cs:20 · AzureTableStorageService.cs:15
    Fix: Validate in Program.cs at startup; throw InvalidOperationException. Sample in the recommended Program.cs above.
  • HIGH-3
    No response validation in PCF High PCF
    The Proff API response is written directly to D365 form fields with no shape validation. If the upstream API or the function is compromised or breaks, malicious or malformed data lands in customer records.
    VirtualControl/Components/Search.tsx (handleSearch)
    Fix: Add a Zod (or similar) runtime schema check on the response; reject on mismatch with a user-visible error.
  • MED-1
    Stale year/month cached in ProffActivityService Medium Backend
    Cached from DateTime.Today at construction. If the worker instance survives across midnight (Premium/Always Ready), counts land on the wrong month row.
    LIVE-ProffCompanyLookup/Services/ProffActivityService.cs:15-16,44-45
    Fix: Compute fresh inside UpdateRequestCountAsync; use DateTime.UtcNow consistently.
  • MED-2
    No try/catch around JSON serialization in HttpHelper Medium Backend
    If a payload throws during serialisation, the function returns a generic 500 instead of a controlled error response.
    LIVE-ProffCompanyLookup/Utils/HttpHelper.cs
    Fix: Wrap in try/catch; return 500 with an error envelope. Better: stop using JsonConvert and let MVC/STJ handle it via the ASP.NET Core integration.
  • MED-3
    Per-request service instantiation in ProffPremiumLookup Medium Backend
    InitializePremiumCacheService and InitializePremiumActivityService new up an AzureTableStorageService on every request (parses connection string, instantiates TableClient). Wasteful, not buggy.
    LIVE-ProffCompanyLookup/ProffPremiumLookup.cs:49,59
    Fix: Inject pre-built table-client singletons via DI as shown in the recommended Program.cs.
  • MED-4
    No user-visible error on failed fetch Medium PCF
    Errors are console.error'd. The user sees an empty result set indistinguishable from "no matches". They retry and burn another API call.
    VirtualControl/Components/Search.tsx (handleSearch catch block)
    Fix: Show a Fluent UI MessageBar at the top of the result area with a retry hint. Log to App Insights via a beacon if available.
  • MED-5
    No request timeout in PCF Medium PCF
    The fetch has no timeout. If the function hangs (cold start > user patience), the request never aborts unless the user retypes.
    VirtualControl/Components/Search.tsx
    Fix: setTimeout(() => controller.abort(), 15000) in handleSearch, cleared on response.
  • LOW-1
    Dead code: ValidationUtils, ProffApiService.GetDetailedCompanyInfo Low Backend
    ValidationUtils.cs is never called. GetDetailedCompanyInfo() (the eniropro variant) is never called and contains an EntityPremiumLicenseIsActive() that always returns false.
